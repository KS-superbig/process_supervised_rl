## 2. 课题想解决什么问题

### 2.1 简单说

现在很多大模型在做长链路推理时，训练信号太少了。

举个例子：一道数学题，模型可能写了 30 步，最后答案错了，那传统训练通常只会告诉它“错了”。但问题是，前 25 步也许是对的，真正出错的是后面几步。只给最后一个“错”字，模型很难知道自己到底哪里做得不好。

所以这个课题想做的事情是：

**不要只看最终答案对不对，还要看推理过程中每一步写得好不好。**

这样模型拿到的反馈会更细、更密，也更容易学到真正有效的推理方式。

### 2.2 技术上怎么说

传统做法更像是：

```text
只根据最终答案是否正确来给 reward
```

这个课题希望变成：

```text
最终答案给一部分 reward
+ 推理过程中的每一步也分别给 reward
```

也就是把“稀疏反馈”变成“稠密反馈”。

## 3. 这个课题最难的地方是什么

### 3.1 奖励黑客（Reward Hacking）

通俗地说，就是模型学会了“刷分”，但没有真正学会推理。

比如我们鼓励它多写步骤，它可能就开始写很多看起来很认真、但其实没有推进解题的废话。这些步骤也许能骗过局部规则，却没有真正帮助得到正确答案。

所以我们不仅要奖励“好步骤”，还要惩罚：

- 重复表达
- 空洞套话
- 没有实质推进的展开

### 3.2 信用分配（Credit Assignment）

一道题最后做对了，不等于每一步都 equally important。

有些步骤只是整理格式，有些步骤才是真正把题做出来的关键转折。于是问题就变成：

**最终成功的“功劳”应该怎么合理分配给中间每一步？**

这是过程监督 RL 里最核心、也最难完全解决的问题之一。

## 5. 第一阶段的总体思路

### 5.1 任务先收敛到数学推理

第一阶段我们先不做开放逻辑题，而是集中在：

- 数学题
- 带明显推理链的题
- 最好能拿到中间解题步骤的题

原因很简单：数学题更适合自动打分。

它通常有：

- 明确的问题
- 标准答案
- 相对可检查的中间步骤

这让我们更容易把过程奖励做出来。

### 5.2 一定要有对照组

我们至少保留两组实验：

1. `final-only reward`
   只看最后答案对不对
2. `final + process reward`
   既看最终答案，也看中间步骤

这样做是为了让实验结果可比较。否则就算最后模型有提升，也很难说明到底是不是“过程奖励”带来的。

### 5.3 过程奖励先拆成四类信号

第一版我们先不把 reward 设计得太复杂，而是拆成下面四类最直观的信号。

#### 1. `step validity`

这一步在数学上是不是成立。

例如：

- 等式变形是否正确
- 代数计算是否有错
- 推导是否明显不合法

#### 2. `step consistency`

这一步有没有和前面已经写过的内容冲突。

例如前面说 `x > 0`，后面又直接在没有解释的情况下按 `x < 0` 推导，这就不一致。

#### 3. `progress contribution`

这一步有没有让解题过程真正往前走。

不是所有“正确的话”都对解题有帮助。我们更希望奖励那些真正缩小问题、推进到结论的步骤。

#### 4. `anti-hacking penalty`

这是专门用来防止模型刷分的惩罚项。

例如：

- 重复说同一件事
- 写了很多套话但没有推导
- 形式上很长，实质上没有信息增量

### 5.4 总奖励怎么理解

第一阶段的总奖励可以简单理解成：

```text
总分 = 最终答案分 + 每一步的过程分
```

写成公式是：

```text
R = α · R_final + β · Σ_t (w_t · r_t)
```

这里你可以这样理解：

- `R_final`：最后答案对不对
- `r_t`：第 t 步写得好不好
- `w_t`：第 t 步在总分里占多大权重

第一阶段里，`w_t` 我们先不让模型自己学，而是先用人工设定的简单方式，比如：

- 每一步权重一样
- 后面关键步骤权重更高

这样做的好处是：先把问题拆小，避免一上来变量太多、最后不知道是哪个设计起作用。

## 6. 数据怎么准备

第一阶段数据可以分成三层来理解。

### 6.1 L1 数据：题目和标准答案

格式大概是：

```text
{问题, 标准答案}
```

这是最基础的一层，用来判断模型最后答得对不对。

### 6.2 L2 数据：题目、步骤和结论

格式大概是：

```text
{问题, 解题步骤, 最终结论}
```

这一层很关键，因为我们做过程监督，需要看到中间步骤。

### 6.3 L3 数据：可以直接喂给 RL 的训练数据

格式大概是：

```text
{问题, 轨迹, 每步 reward, 最终 reward}
```

这里的“轨迹”就是模型完整解一道题的一条推理过程。

## 7. 第一阶段建议用什么数据

为了兼顾“尽快跑通”和“后续可扩展”，建议先走双数据源路线：

### 第一优先级：`GSM8K`

适合用来快速跑通流程，原因是：

- 数据干净
- 难度适中
- 很多题自带较清晰的中间解法

适合做第一轮系统联调。

### 第二优先级：`PRM800K`

这个数据集和“过程监督”主题非常贴近。

因为它本身就包含大量与步骤质量相关的标注，非常适合后续：

- 校验我们的规则打分器
- 训练更强的过程奖励模型
- 替换掉纯规则方案

所以比较稳妥的策略是：

**先用 GSM8K 跑通，再用 PRM800K 把过程监督做深。**

## 8. 第一阶段要做的 8 件事

### Step 1. 选数据、拿数据

目标是先拿到题目和答案，最好还能拿到中间步骤。

第一步不要求数据最完美，只要求能让我们尽快开始。

### Step 2. 清洗数据、统一格式

把不同来源的数据都整理成统一 schema，方便后面共用脚本。

这一步通常包括：

- 去重
- 清理坏样本
- 划分 train / dev / test
- 锁定 test 集，避免后面污染

> 当前状态：这一步已经完成基础版本，现有脚本已经可以在远端稳定生成统一格式数据。
